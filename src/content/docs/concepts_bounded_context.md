## 解決する課題

「顧客」という言葉が、チームによって意味が違う。

- 販売チーム：「購入者」
- サポートチーム：「チケットを上げる人」
- 請求チーム：「支払い義務のある法人」

これを1つのモデルで表現しようとすると、`Customer` クラスに全員の都合が詰め込まれて壊れる。

## 概念

**境界づけられたコンテキスト（Bounded Context）＝「翻訳不要ゾーン」**

境界の内部では：
- ユビキタス言語が一貫して使われる
- 同じ言葉は1つの意味しか持たない
- チームが自律的にモデルを変更できる

境界の外では同じ言葉が別の意味を持ってよい。それぞれのコンテキストが自分の「顧客」を独自に定義する。

## サブドメインとの違い

| | サブドメイン | 境界づけられたコンテキスト |
|---|---|---|
| 何を表すか | 問題空間（業務の分割） | 解決空間（コードの境界） |
| 誰が決めるか | ドメインエキスパート | 開発チーム |
| 例 | 「注文管理という業務領域」 | 「注文サービスというコードの塊」 |

1対1に対応しないことがある。1つのサブドメインに複数のコンテキストが存在することも、その逆もある。

## なぜ重要か

単一の巨大なモデルを全システムで共有すると：
- `Customer` に全部門の属性が詰め込まれ肥大化する
- チーム間の変更が影響し合い、リリースが困難になる
- 用語の意味が揺れ、コミュニケーションコストが増大する

Bounded Contextを明示的に設けることで、**各チームが自律的にモデルを進化させられる**。

## 適用場面

- マイクロサービスの境界を決める際（1コンテキスト≒1サービスが目安）
- 他チームとの統合方法を決める（コンテキストマップを描く）
- 「このモデルはここまで」という責任範囲を明示するとき
